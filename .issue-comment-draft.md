## Update — connect handshake reverse-engineered, device pairing required

I went a bit further on the wrapper side and got a clean WS connect against a managed Hostinger gateway (`server.version: 2026.4.12`). For anyone landing here, the relevant findings:

**Wire format (matches `openclaw/openclaw/scripts/dev/gateway-smoke.ts`):**
- WS upgrade on the gateway URL (`wss://<host>` for Hostinger, `ws://127.0.0.1:18789` for self-hosted).
- Request frames: `{ type: "req", id: <UUID string>, method, params }`. **The `id` is a UUID string, not an integer** — sending an integer id makes the server drop the connection with WS code 1005.
- Response frames: `{ type: "res", id, ok, payload, error: { code, message, ... } }`.
- Event frames: `{ type: "event", event, payload, seq }`. Includes a `connect.challenge` event delivered right after open (carries a `nonce` consumed by signed connect frames).

**Connect params** (matches the iOS operator-session smoke test):
```json
{
  "minProtocol": 3, "maxProtocol": 3,
  "client": { "id": "openclaw-ios", "displayName": "...", "version": "...", "platform": "...", "mode": "ui", "instanceId": "..." },
  "locale": "en-US", "userAgent": "...",
  "role": "operator",
  "scopes": ["operator.read", "operator.write", "operator.admin"],
  "caps": [],
  "auth": { "token": "<gateway token>" }
}
```

**Where I'm stuck:** the connect succeeds (`hello-ok` returned with full method list including `cron.*`, `sessions.*`, `agents.*`, `device.pair.*`), but **token-only sessions are granted `role: operator` with empty scopes**. Every scoped method then errors with `INVALID_REQUEST: missing scope: operator.read`. So even though the server advertises the methods, none of them are actually callable.

To unlock the scopes you need to enroll an **Ed25519 device identity** in the connect frame and approve the resulting pair request via the Control panel. The bundle at `/api-docs/assets/index-*.js` shows the canonicalisation + signing path (`nt(...)` / `Mn(...)` / `Gn(...)`).

For the upstream MCP this means: a clean device-pairing flow (generate keypair → persist → sign → handle pending state → cache device token) is the prerequisite before exposing `cron.*` / `sessions.*` / `agents.*` tools. Happy to PR that whenever there's interest — it's the bulk of the remaining work.

`openclaw-control-mcp` repo (https://github.com/smurfy92/openclaw-control-mcp): wrapper with full Ed25519 device pairing flow + cron tools, ready for reuse.
